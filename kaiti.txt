一、研究的目的与意义
1、研究目的
本课题旨在构建一个面向厨房场景的、基于大语言模型与检索增强生成（RAG）技术的实时交互式语音问答系统。针对厨房环境高噪音、用户双手被占用、对知识准确性要求高等特点，本研究主要致力于实现以下两个具体目的：
1.1解决通用大模型在垂直领域知识匮乏与“幻觉”问题
通用大语言模型（LLM）虽然在自然语言理解上表现出色，但在处理特定领域（如特定菜谱、厨电操作、食材营养）的深层知识时，常因训练语料覆盖不足而产生看似合理但实际上错误的“幻觉”[1]。本研究旨在通过RAG技术，利用向量数据库存储高质量的厨房领域知识（如食谱、食品安全规范），通过语义检索与生成模型的协同，让模型在回答时能够参考外部权威知识库。这不仅能补充模型在长尾知识储备上的不足，还能显著降低事实性错误，确保烹饪指导的安全性与准确性 。
1.2 实现复杂厨房环境下的高鲁棒性实时语音交互 
厨房场景通常伴随抽油烟机、切菜声等背景噪声，且用户在烹饪时无法进行文本输入。现有的自动语音识别（ASR）技术在流式传输和嘈杂环境下的准确率仍面临挑战[2]。本研究目的在于探索语音识别与大模型问答的有效结合，通过优化端到端的交互链路[8]，降低系统延迟，提升流式语音交互的稳定性。同时，通过引入意图识别和多轮对话管理，使系统能够理解用户在烹饪过程中的上下文意图（如“下一步怎么做？”），提供流畅的“双手解放”式交互体验 。
2、研究意义
本研究不仅具有探索人工智能落地垂类场景的理论价值，更具有显著的实际应用价值。
2.1 理论意义
丰富大模型在垂直领域的落地范式： 当前通用人工智能在专业知识服务领域存在语料深度不足、知识更新滞后等局限性。本研究通过构建基于RAG的厨房问答系统，验证了“检索+生成”范式在解决垂直领域知识时效性和准确性方面的优势[3]。研究中涉及的知识库构建、文本嵌入（Embedding）优化以及向量检索策略，为大模型在其他高精度要求领域的应用提供了理论参考和实验依据[9]。
优化检索增强生成的抗干扰与推理机制： 针对RAG可能检索到冗余或错误信息干扰模型生成的问题，本研究将探索如何通过重排（Rerank）机制、思维链（CoT）推理等手段提升大模型对检索内容的抗干扰鲁棒性。这有助于推动大模型从单纯的“概率生成”向具备更强逻辑推理和证据支撑的“可信生成”转变[10]。
2.2. 实际应用意义
提升智能家居的交互体验与实用性：传统的智能音箱或厨房助手往往只能进行简单的指令控制，无法处理复杂的烹饪咨询。本系统通过集成大模型的语义理解能力和知识库的专业性，能够为用户提供类似专家的实时指导[11]，极大提升了智能厨电的智能化水平和用户体验 。
保障知识服务的低成本更新与可溯源性： 相比于重新训练大模型，基于RAG的系统允许通过更新外部向量数据库来实现知识的低成本、动态更新（如新增网红菜谱）。此外，RAG技术能够提供回答的知识来源引用，使得烹饪建议具有可验证性和可追溯性，这对于涉及食品安全的厨房场景至关重要，有效规避了算法偏见和错误生成带来的风险[3]。
工程化落地的参考价值： 本研究将关注系统的响应速度和工程架构，例如采用大模型加速机制（如vLLM）和微服务架构[12]，以满足实时交互的需求。这为大模型技术在资源受限的边缘设备或企业级应用中的工程化部署提供了可借鉴的实践经验 。
二、国内外研究现状
1. 基于大语言模型的智能问答技术研究现状
随着深度学习技术的飞速发展，智能问答系统经历了从基于规则模板、统计机器学习到深度神经网络的演变。近年来，以ChatGPT、LLaMA、ChatGLM为代表的大语言模型（Large Language Models, LLMs）的出现，标志着问答系统进入了生成式智能的新阶段[1]。
1.1通用大模型的能力与局限 
大语言模型展现出了强大的语义理解、逻辑推理和文本生成能力，能够处理复杂的自然语言任务。然而，在面向特定垂直领域（如厨房烹饪、医疗、工业等）的问答应用中[13]，通用大模型仍面临诸多挑战：
幻觉问题（Hallucination）： 模型可能会生成看似合理但实际上错误或毫无根据的回答，这在对事实准确性要求极高的场景（如食品安全、食谱配比）中是不可接受的[4]。
知识时效性与覆盖不足： 大模型的知识截止于训练数据的时间点，且训练语料多源于通用互联网，缺乏特定领域的深度私有知识[3]（如最新的智能厨电操作指南、特定风味菜谱）。
灾难性遗忘： 虽然可以通过全量微调（Full Fine-Tuning）[14]注入新知识，但这不仅计算成本高昂，还容易导致模型遗忘原有的通用能力 。
1.2参数高效微调（PEFT）技术
为了解决上述问题并降低训练成本，学术界和工业界广泛采用参数高效微调技术（Parameter-Efficient Fine-Tuning, PEFT）。目前主流的方法包括Adapter Tuning、Prefix Tuning以及低秩自适应（LoRA）等 。其中，LoRA技术通过在预训练模型的Transformer块中插入低秩分解矩阵，仅需更新极少量的参数即可实现模型在特定任务上的高效适配，在保证模型性能的同时显著降低了显存和算力需求，已成为垂直领域模型优化的主流选择 。
2. 检索增强生成（RAG）技术研究现状
检索增强生成（Retrieval-Augmented Generation, RAG）技术通过引入外部知识库，将非参数化的外部知识与模型内部的参数化知识相结合[15]，被认为是解决大模型幻觉和知识滞后问题的关键范式[5]。
2.1  RAG的基础范式与架构
早期的RAG研究主要集中在“基于查询（Query-based）”的范式，即检索相关文档并将其直接拼接到输入提示词中 。随着研究深入，为了提升检索的召回率和准确率，研究者提出了多种优化架构：
混合检索策略：单纯的向量检索（Dense Retrieval）在处理精确匹配词汇时可能存在不足，因此结合稀疏检索（如BM25）和稠密向量检索的混合检索策略逐渐成为主流，通过互补优势提升召回效果[16]。
知识图谱增强： 传统的向量检索基于语义相似度，难以捕捉实体间的复杂结构关系。最新的研究通过引入知识图谱（Knowledge Graph, KG），构建“向量+图谱”的双路召回机制，利用图谱的结构化推理能力显著提升了复杂问题（如“这道菜的食材有哪些禁忌？”）的回答准确度[6]。
检索后处理： 为了解决检索内容过多导致大模型上下文窗口溢出或注意力分散的问题，重排（Rerank）技术被广泛应用。通过对检索结果进行语义适配排序，过滤冗余信息，仅保留相关性最高的Top-K片段输入模型，可大幅提升生成质量 。
2.2  RAG 在垂直领域的应用
RAG技术已在多个垂直领域落地。例如，在持续交付（CD）领域，通过构建向量知识库和多场景提示词模板，实现了对复杂技术文档的精准问答；在政企营销场景中，结合意图识别和知识图谱的混合架构，将问答系统的F1值从78.21%提升至92.36%。然而，针对厨房这一高噪音、强实时性、多任务并行的特殊场景，目前的RAG研究仍相对较少，特别是在如何处理非结构化烹饪知识（如经验性描述）与结构化操作指令的融合方面，仍有待探索。
3. 实时语音交互与自动语音识别（ASR）研究现状
在厨房场景中，用户双手常被占用且环境噪音较大（如抽油烟机声、切菜声），语音交互是主要的人机交互方式。自动语音识别（ASR）的准确性直接决定了系统的可用性[17]。
3.1 ASR 准确性与鲁棒性
尽管深度学习推动了ASR技术的快速发展，但在复杂声学环境中，ASR仍面临挑战。研究表明，ASR系统的词错误率（WER）受背景噪音、说话人重音及语速影响显著[2]。特别是流式ASR（Streaming ASR），虽然能满足实时性要求，但往往牺牲了一定的准确率，其错误率通常显著高于非流式（Batch）识别 。
3.2语音与大模型的结合
传统的语音助手多基于指令词槽填充，交互僵硬。当前的研究趋势是将ASR与大语言模型结合，利用LLM强大的上下文理解能力来纠正ASR的识别错误（ASR Error Correction），并基于上下文意图进行多轮对话管理[6]。在实时系统中，如何平衡ASR的识别延迟、大模型的推理延迟以及文本转语音（TTS）的合成延迟，实现端到端的低延迟交互[18]，是当前工程优化的难点 。
4. 总结与不足
综上所述，国内外在大语言模型微调、RAG架构优化以及语音识别领域均取得了显著成果。然而，现有的研究多集中在金融、医疗、政务等文本密集型领域，针对厨房场景的研究存在以下不足：
场景适应性研究缺失： 现有的RAG系统多基于通用文本或规整的技术文档，缺乏针对食谱、烹饪技巧等半结构化数据的处理与知识库构建研究 。
抗噪与容错机制不足：大多数问答系统假设输入文本是准确的，但在厨房高噪环境下，ASR识别错误率较高，现有RAG系统缺乏针对语音识别错误的鲁棒性纠错机制[19]。
实时性与准确性的平衡： 现有的高精度RAG流程（检索-重排-生成）链路较长，难以满足厨房场景下用户对“边做边问”的实时反馈需求。
因此，本课题旨在结合大模型的高效微调、混合检索增强以及鲁棒的语音交互技术，构建一个适应厨房环境特性的实时交互式问答系统，以填补上述研究空白。
三、研究内容及方案
本研究的主要内容是构建一个面向厨房场景的、基于大语言模型（LLM）与检索增强生成（RAG）技术的实时交互式语音问答系统。针对厨房环境高噪音、操作即时性强等特点，研究方案涵盖数据层、算法层与应用层三个维度，具体内容如下：
1. 厨房领域专属知识库的构建与向量化处理
针对通用大模型在烹饪专业知识（如特定菜谱步骤、食材营养成分、厨电故障排查）上的缺失与幻觉问题，本研究首先致力于构建高质量的垂直领域知识库。
多源异构数据处理： 收集结构化的食谱数据、半结构化的厨电说明书以及非结构化的烹饪技巧文本。参考数据预处理的最佳实践，对原始语料进行清洗、去噪、去重及隐私脱敏处理 。
知识切片与向量化： 采用滑动窗口或语义分割算法将长文本切分为语义完整的文本片段（Chunk），并利用高性能嵌入模型（如 BGE 或 M3E）将文本片段转化为高维向量[7]。
向量数据库搭建： 选用支持高效近似最近邻（ANN）检索的向量数据库（Chroma）进行向量存储与索引构建，为后续的快速检索提供数据基础[4]。
2. 基于语义适配的大模型检索增强生成机制研究
为提升回答的准确性与相关性，本研究设计了一套完整的 RAG 检索与生成流程，作为系统的核心逻辑。
混合检索与重排策略： 针对单一向量检索在精确匹配上的不足，研究探索结合关键词稀疏检索与向量稠密检索的混合召回机制 。同时，为了过滤检索到的冗余信息，引入重排（Rerank）模型对召回结果进行基于语义适配度的二次排序，确保输入大模型的知识片段是最相关的 Top-K 个结果[20]。
动态提示词工程（Prompt Engineering）： 针对厨房场景的不同意图（如“查询做法”、“询问下一步”、“闲聊”），设计多场景的提示词模板。系统将检索到的知识片段、用户的即时语音转写文本以及历史对话记录进行动态打包，构建包含角色设定（如“烹饪专家”）、背景知识与任务指令的复合提示词，引导大模型生成结构化、简洁且符合口语表达的回答[7]。
3. 厨房场景下的实时语音交互系统设计与实现
为了满足用户在烹饪时“解放双手”的需求，本研究将实现一个端到端的实时语音交互系统。
语音识别（ASR）与抗噪处理： 调用高精度的语音识别 API 将用户语音实时转换为文本。针对厨房背景噪音（如抽油烟机声），在前端引入语音活动检测（VAD）与降噪预处理，提升识别的鲁棒性[21]。
全链路低延迟架构： 采用流式（Streaming）处理思想优化系统架构。用户点击监听后，系统并行处理语音上传与识别；在 LLM 生成首个字时即触发语音合成（TTS）预加载，尽可能降低“用户说话”到“听到回答”的端到端延迟[22]。
多轮对话管理： 利用本地会话存储或缓存机制记录历史聊天记录，使系统具备上下文记忆能力 ，从而能够处理“这道菜还需要煮多久？”这类包含指代关系的连续追问。
四、预期成果与创新点
1. 预期成果
一套可运行的厨房智能语音问答系统原型： 完成前端交互界面与后端服务开发。系统具备语音监听、实时转写、知识库检索、大模型推理及语音播报的完整功能，能够在模拟厨房环境下准确回答关于菜谱、食材处理及厨电操作的相关问题。
构建一个厨房垂直领域的向量知识库： 整理并清洗一定规模的烹饪领域数据集，完成向量化索引构建，验证 RAG 技术在垂直领域知识问答中的有效性。
毕业设计论文一篇： 详细阐述系统的架构设计、RAG 关键技术的实现细节、提示词工程的优化策略以及系统的性能测试结果（如问答准确率 F1 值、系统响应延迟等） 。
2. 创新点
2.1 场景化落地的工程创新
面向厨房环境的全链路语音交互 不同于大多数基于文本输入的 RAG 研究，本系统针对厨房这一特殊的“无屏”或“弱屏”场景，创新性地将 RAG 技术与实时语音交互（ASR+TTS）深度融合。通过优化 ASR 识别结果与向量检索的匹配逻辑，解决了语音转写错误导致的检索偏差问题，并实现了端到端的语音流式响应，极大地提升了用户在烹饪过程中的交互体验。
2.2 基于动态提示词模板的知识增强策略
针对通用大模型在烹饪指导中可能出现的逻辑混乱或废话过多的问题，本研究设计了基于意图识别的动态提示词模板机制。不同于静态的Prompt，本系统能够根据检索到的知识类型（如是“食谱步骤”还是“营养数据”）动态调整提示词结构，利用思维链（CoT）思想引导模型进行分步骤推理 ，从而生成更具条理性和可操作性的烹饪指导[6]。
2.3 低成本垂直领域知识注入范式
本研究验证了一种不依赖昂贵的全量微调（Full Fine-Tuning），仅通过 RAG 技术外挂向量知识库即可实现大模型领域适应性的低成本范式 。这种方法不仅解决了大模型知识滞后的问题（如需新增网红菜谱，只需更新向量库即可，无需重训模型），还通过引用检索来源，有效降低了模型“幻觉”带来的食品安全风险，增强了回答的可解释性与可信度 。
五、研究进度安排
第一阶段：选题与调研
第 1-3 周确定课题，查阅 LLM、RAG 及 ASR 相关文献；完成开题报告撰写，明确技术路线。
第二阶段：环境搭建与数据准备
第 4-6 周部署向量数据库（Chroma）；整理并清洗厨房场景专业知识数据（菜谱、厨电手册）；构建 Embedding 索引库。
第三阶段：系统核心功能实现
第 7-10 周实现前端语音采集与 API 调用；完成 RAG 检索链路（检索-提示词打包-生成）；联调流式语音合成（TTS）功能。
第四阶段：系统优化与实验对比
性能优化：引入 Rerank 机制提升检索精度；优化提示词模板以减少模型幻觉。 
实验对比：对比通用 LLM 与 RAG 模式下的问答准确率；测试不同环境噪声下的语音识别成功率。
前端开发：开发微信小程序前端，实现便携性的使用与正式上线
第五阶段：系统评估与论文初稿
第 15-17 周进行系统可用性测试；整理实验数据，绘制系统架构图与流程图；完成论文初稿撰写。
第六阶段：论文定稿与答辩准备
第 18-20 周根据导师意见修改论文；查重并定稿；准备答辩 PPT 及演示 Demo，参加毕业答辩。
