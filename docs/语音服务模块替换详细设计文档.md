# 语音服务模块替换详细设计文档（STT/TTS 可插拔）

## 0. 背景与目标

本项目当前语音链路为“前端采集 PCM16 → 后端 WebSocket → STT 识别 → LLM 回复 → TTS 合成 → 前端播放”。现阶段需要先打通**本地轻量模型**的闭环（降低第三方 API 依赖、增加可量化工作量），并为后续增加“服务切换”前端界面做好预留：后续只通过修改配置，即可在**本地轻量模型 / 云端大模型 / 第三方 API**之间无缝切换，且不影响上层业务与前端交互协议。

### 0.1 范围（本次交付）

1. 设计高度模块化的语音服务层，支持“热插拔”式替换，业务代码零侵入（或最小侵入：只替换调用点）。
2. 选型并验证 1 款本地可跑的轻量 STT 与 TTS 模型：
   - CPU 实时推理 ≤ 1× 实时因子（RTF ≤ 1）
   - 内存占用 ≤ 1 GB（常驻内存）
3. 制定统一语音服务抽象接口，覆盖五大生命周期：初始化、推理、资源释放、健康检查、指标上报。
4. 建立配置驱动的服务注册表（JSON），通过环境变量或启动参数动态加载指定模型，无需改动代码即可切换。
5. 输出本文档（含选型表、分层架构、接口定义、集成步骤、切换流程、CI/CD 钩子、交付清单）。

### 0.2 非范围（本次不做）

- Web 前端“服务切换”界面实现（仅预留后端配置能力）。
- GPU 大模型（如 MOSS‑TTS‑Realtime、FunASR 大模型等）的实际部署与性能调优（仅作为可插拔 Provider 预留）。
- 训练/微调（本阶段只做推理集成与工程化）。

## 1. 现状梳理（与改造点）

### 1.1 当前实现概览

- 后端语音会话入口：`backend/api/server.py` 的 `/ws/voice` WebSocket
- STT：`backend/stt/asr_client.py`（第三方 WS 协议）+ `server.py` 内部回调触发回复
- TTS：`backend/tts/tts_client.py`（第三方 WS 协议）+ `server.py` 内部流式推送 PCM16
- 配置：`backend/config.py` 读取 `.env` / 环境变量

### 1.2 当前问题

- STT/TTS 与业务耦合在 `server.py`：替换模型时需要修改业务逻辑与会话流程。
- 只有“第三方 API Provider”，缺少“本地模型 Provider”与统一生命周期接口。
- 缺少统一的健康检查、指标采集、性能基准与回归测试入口。

### 1.3 改造目标（工程口径）

- 业务层只依赖抽象接口（STT/TTS Protocol），不依赖具体模型或厂商实现。
- Provider 通过注册表配置加载：`STT_PROVIDER`/`TTS_PROVIDER` 指向一个 provider_id。
- 同一套前端协议不变：
  - 上行音频：PCM16、16k、单声道，按 chunk 发送
  - 下行音频：PCM16 chunk 二进制 + `tts_start` 事件携带采样率

## 2. 本地模型选型（轻量闭环）

本节给出“本次闭环”的推荐组合，并提供对比维度与验证方法。注意：WER/CER/RTF/内存等指标需在本项目的目标机器上用统一脚本跑出结果，填入表格并纳入 CI 阈值。

### 2.1 选型结论（本次建议）

- **STT（主线）**：sherpa-onnx Streaming ASR（ONNX Runtime，CPU 友好，低延迟）
- **TTS（主线）**：Piper TTS（ONNX Runtime，CPU 友好，小模型、易部署）

选这两类的核心原因：
- 都是“本地推理、无云依赖”的典型路线，符合“减少 API 调用、增加工程工作量”的目标。
- ONNX Runtime 在 CPU 上通常更易达到 RTF ≤ 1，且常驻内存可控。
- 未来升级到 GPU 大模型时，只需要新增 Provider 适配层，不影响业务。

### 2.2 模型对比表（模板）

| 模块 | 候选 | 许可证/合规 | 语言 | 体积（权重） | 依赖 | 流式 | 典型延迟 | RTF（CPU） | 常驻内存 | 备注 |
|---|---|---|---|---:|---|---|---:|---:|---:|---|
| STT | sherpa-onnx streaming（Zipformer/Paraformer onnx） | 以仓库/模型为准 | 中/英/方言（随模型） | 待测 | onnxruntime | 支持 | 待测 | 待测（目标 ≤1） | 待测（目标 ≤1GB） | 工程落地快 |
| STT | FunASR Paraformer-streaming（PyTorch） | 以仓库/模型为准 | 中文强 | 待测 | torch | 支持 | 可到 480–600ms chunk | 待测 | 待测 | 适合后续“高质量版” |
| TTS | Piper（onnx） | 以仓库/模型为准 | 依具体 voice | 待测 | onnxruntime | 可做伪流式（分句） | 待测 | 待测（目标 ≤1） | 待测（目标 ≤1GB） | 轻量易部署 |

### 2.3 验证标准（必须可复现）

- 音频输入规范：PCM16、16kHz、mono
- STT 指标：
  - 首字延迟（ms）
  - 句子结束延迟（ms）
  - RTF（处理耗时 / 音频时长）
  - CER（中文字符错误率），或 WER（词错误率）
  - 常驻内存（RSS）与峰值内存
- TTS 指标：
  - 首包音频延迟（ms）
  - RTF（合成耗时 / 音频时长）
  - 音质主观评测（可用 MOS 小样本）
  - 常驻内存（RSS）与峰值内存

## 3. 模块分层架构（可插拔）

### 3.1 分层说明

- **接口层（Interface）**：定义 STT/TTS 抽象协议与统一数据结构（输入输出 schema、异常规范）
- **适配层（Adapter）**：把“具体模型/厂商 SDK”包装成统一接口实现
- **模型运行时层（Runtime）**：负责模型加载、推理、缓存管理、线程/异步策略、资源释放
- **配置层（Config/Registry）**：注册表 + 环境变量解析 + provider 构造
- **监控层（Observability）**：健康检查、指标上报、日志、性能基准

## 4. 统一接口定义（可直接落地）

本节给出“方法签名 + 输入输出 Schema + 异常规范 + 同步/异步模式”。

### 4.1 统一数据结构（Schema）

#### 4.1.1 音频块

- `AudioChunk`
  - `pcm16_bytes: bytes`：PCM16 little-endian
  - `sample_rate: int`：默认 16000
  - `channels: int`：默认 1
  - `is_final: bool`：是否为一句/一段的最后一块（流式端点时置 True）
  - `timestamp_ms: int | None`：可选，采集端时间戳

#### 4.1.2 STT 结果

- `SttResult`
  - `text: str`
  - `is_final: bool`：是否为最终结果
  - `confidence: float | None`
  - `start_ms: int | None` / `end_ms: int | None`：可选时间戳
  - `raw: dict | None`：适配层透传（便于调试，不影响业务）

#### 4.1.3 TTS 输出

- `TtsChunk`
  - `pcm16_bytes: bytes`
  - `sample_rate: int`：建议统一下行 24000 或 16000（由 provider 决定，但需对前端透明：通过 `tts_start.rate` 告知）
  - `is_final: bool`

### 4.2 生命周期接口（Protocol/ABC）

建议用 Python `typing.Protocol`（便于注入与测试），或 `abc.ABC`（更强约束）。接口需覆盖：初始化、推理、释放、健康检查、指标上报。

#### 4.2.1 通用能力

- `initialize() -> None`：加载模型/建立连接（必须幂等）
- `close() -> None`：释放资源（幂等）
- `health_check() -> dict`：返回健康状态（可含版本、加载状态、最近错误）
- `collect_metrics() -> dict`：返回关键指标（延迟、RTF、错误计数、内存等）

异常规范：
- 任何 Provider 内部异常不得直接冒泡到 WS 发送线程导致崩溃；统一封装为：
  - `VoiceServiceInitError`
  - `VoiceServiceRuntimeError`
  - `VoiceServiceConfigError`

#### 4.2.2 STT 接口（流式）

- `create_session(session_id: str) -> None`
- `accept_audio(session_id: str, chunk: AudioChunk) -> None`
- `poll_result(session_id: str) -> list[SttResult]`
  - 返回“自上次 poll 以来产生的结果”（可包含 partial 与 final）
- `reset_session(session_id: str) -> None`

同步/异步建议：
- `accept_audio()` 必须是轻量操作：只入队/写入流，避免阻塞 WS receiver。
- 真实推理在后台线程/协程执行，`poll_result()` 取队列。

#### 4.2.3 TTS 接口（可中断流式）

- `synthesize_stream(session_id: str, text: str) -> Iterator[TtsChunk]`
  - 也可设计为 `async def` 返回异步生成器，便于直接推送 WS 二进制
- `stop(session_id: str) -> None`
  - 立即停止当前合成并释放/复位内部状态（用于“可打断”）

## 5. 配置驱动注册表（JSON）

本节定义“注册表 Schema + 环境变量 + 动态加载策略”，确保“只改配置即可切换 Provider”。

### 5.1 环境变量（建议）

- `VOICE_REGISTRY_PATH`：注册表文件路径（默认 `backend/voice/registry.json`）
- `STT_PROVIDER`：STT provider_id（如 `local_sherpa_zipformer_small`）
- `TTS_PROVIDER`：TTS provider_id（如 `local_piper_zh`）
- `VOICE_METRICS_ENABLE`：是否启用指标采集（默认 true）
- `VOICE_BENCH_ENABLE`：是否允许访问基准测试接口（默认 false，避免生产暴露）

### 5.2 注册表结构（示例）

```json
{
  "stt": {
    "local_sherpa_zipformer_small": {
      "type": "sherpa_onnx",
      "model_dir": "models/stt/sherpa-zipformer-small-zh",
      "sample_rate": 16000,
      "num_threads": 4,
      "endpoint": {
        "enabled": true,
        "rule1_min_trailing_silence": 0.8,
        "rule2_min_trailing_silence": 1.2,
        "rule3_min_utterance_length": 200
      }
    },
    "api_volc_ws": {
      "type": "volc_ws",
      "endpoint": "wss://...",
      "env": {
        "ASR_APP_ID": "ASR_APP_ID",
        "ASR_ACCESS_TOKEN": "ASR_ACCESS_TOKEN"
      }
    }
  },
  "tts": {
    "local_piper_zh": {
      "type": "piper",
      "model_path": "models/tts/piper/zh.onnx",
      "config_path": "models/tts/piper/zh.json",
      "sample_rate": 22050,
      "speaker_id": 0,
      "length_scale": 1.0,
      "noise_scale": 0.667
    },
    "api_tencent_ws": {
      "type": "tencent_ws",
      "env": {
        "TC_APP_ID": "TC_APP_ID",
        "TC_SECRET_ID": "TC_SECRET_ID",
        "TC_SECRET_KEY": "TC_SECRET_KEY"
      }
    }
  }
}
```

说明：
- `type` 决定加载哪个 Adapter。
- `env` 用于声明“从哪些环境变量读密钥”，注册表不存放敏感信息。
- 本地模型路径建议放在 `models/`，并保持 `.gitignore` 忽略，避免仓库膨胀与泄密。

### 5.3 动态加载策略

- 服务启动时：
  1. 读取 `VOICE_REGISTRY_PATH`
  2. 读取 `STT_PROVIDER` / `TTS_PROVIDER`
  3. 构造对应 Adapter 并 `initialize()`
- 切换 Provider：
  - 修改环境变量或注册表后重启服务即可生效（本阶段不做运行时热切换，避免状态复杂度）

## 6. 本地模型集成步骤（可执行）

本节提供“环境准备 → 权重下载 → 推理 smoke test → 单元测试 → 性能基准”的落地步骤。各模型的具体下载地址与许可证需在实施时确认并记录。

### 6.1 环境准备

- Python：建议 3.10+
- 依赖：
  - onnxruntime（CPU 版）
  - sherpa-onnx（若选用）
  - piper-tts（若选用）或对应 Python 绑定

注意：本项目现有 `backend/requirements.txt` 未包含以上依赖；建议在“语音服务层”落地时新增可选依赖，并在文档中标注其版本范围与安装方式。

### 6.2 权重下载与存放规范

- 统一目录：
  - `models/stt/<provider>/...`
  - `models/tts/<provider>/...`
- 不提交到 Git：
  - `models/` 已在 `.gitignore` 中忽略
- 必须随交付提供：
  - 每个模型的 LICENSE 文件或链接（写入交付清单）
  - 模型来源、版本号、哈希校验（用于“模型签名校验”）

### 6.3 推理脚本（文档内提供模板）

#### 6.3.1 STT（流式）模板

- 输入：1 段 WAV（16k/mono），按 200ms 切块模拟实时
- 输出：partial + final 文本与延迟统计

#### 6.3.2 TTS 模板

- 输入：固定文本集（短句/长句/数字单位/中英混合）
- 输出：首包延迟、RTF、产出音频长度与播放验证

### 6.4 单元测试（建议）

- `test_stt_smoke`: 固定短音频 → 结果包含关键关键词（弱断言，避免过拟合）
- `test_tts_smoke`: 固定短文本 → 输出 PCM 非空、采样率正确
- `test_interrupt`: 合成中 stop → 输出停止且资源释放

### 6.5 性能基准脚本（必须）

基准脚本需输出 JSON 结果并可被 CI 解析，至少包含：
- `rtf`
- `first_token_latency_ms`（STT 可用“首个 partial”）
- `first_audio_latency_ms`（TTS）
- `rss_mb` / `peak_rss_mb`

通过阈值策略拦截：
- RTF > 1 直接 fail（可允许 `rtf <= 1.2` 的灰度阈值，逐步收紧）
- 内存 > 1024MB fail

## 7. 服务切换流程（配置变更→验证→回滚）

### 7.1 切换流程

1. 修改配置：`STT_PROVIDER`/`TTS_PROVIDER` 或 `registry.json`
2. 重启服务：加载新 Provider
3. 灰度验证：
   - 运行 smoke test（本地或 CI）
   - 跑基准脚本，比较 RTF/延迟/内存
4. 回滚策略：
   - 恢复上一版 provider_id
   - 回滚 registry.json（如变更过）
   - 重启服务并复测

### 7.2 与前端协议的兼容约束

- 上行音频格式不变：PCM16/16k/mono
- 下行音频格式允许不同采样率，但必须通过 `tts_start.rate` 明确告知前端
- 可打断机制不变：一旦检测到用户开口，立刻 `tts.stop()` 并清空待播队列

## 8. CI/CD 钩子（建议落地方式）

### 8.1 自动语音回归测试

- 每次合并触发：
  - STT/TTS smoke test
  - 关键接口兼容性检查（schema 校验）

### 8.2 性能阈值拦截

- 基准脚本在 CI 中跑（可用 CPU runner）

### 8.3 模型签名校验

- 对本地模型文件生成 sha256 清单
- 启动时校验（可配置开关）
- 防止模型被替换导致结果漂移或引入风险

## 10. 风险与对策

- **CPU 实时性不达标**：优先调小模型、降低 chunk 大小、减少后处理；必要时把 STT/TTS 拆到独立服务或迁移到 GPU。
- **音频格式不一致**：统一在接口层做重采样与声道归一（或强约束前端只送 16k mono）。
- **模型/依赖冲突**：建议为“语音推理”单独 venv 或按 extra requirements 管理。
- **端点检测不稳定**：将 endpoint 参数纳入 registry，并提供 debug 日志与可视化（本次只预留）。

## 11. 附录：推荐默认参数（与现有前端一致）

- 采样率：16000
- 通道：1
- chunk：200ms（与现有前端分块一致）
- STT endpoint：
  - trailing_silence：0.8–1.2s（按体验调）
- TTS 播放：
  - 优先使用 provider 原生采样率并通过 `tts_start.rate` 告知前端
