# 打断功能快停与整句保留设计

## 1. 背景

当前项目的打断功能已经“基本可用”，但仍存在两个核心问题：

1. 打断灵敏度不稳定  
   - 有时用户未明显开口，系统会提前停播。
   - 有时用户已经开口，TTS 仍继续播出一小段。

2. 打断后的用户语句不完整  
   - 用户说一句完整的话去打断时，ASR 最终结果常只保留后半句。
   - 现象本质不是 LLM 问题，而是“停播时机”和“用户开口前几百毫秒音频”没有被正确保留。

用户已明确当前调优优先级：

- 优先级 1：尽快停播
- 优先级 2：尽量保住整句
- 可以接受：误打断略有增加
- 不接受：通过额外等待窗口来换识别完整度
- 提交策略：`partial` 只用于打断和展示，`final` 才提交给 LLM

因此，本次设计目标不是继续单独调 detector 阈值，而是把“更早停”和“补回前半句”这两件事在链路上做对。

## 2. 目标

本次设计的目标如下：

1. 用户一开口时，TTS 尽快停播。
2. 停播后，用户打断时说的整句尽量完整进入 STT `final`。
3. 仍保持 `final -> LLM` 的提交策略，不引入 partial 直喂 LLM。
4. 保留现有动态打断算法作为论文主算法与运行时主判定依据。
5. 修改范围尽量收敛，不引入第二套 STT 或客户端主导的打断状态机。

非目标：

- 不做 partial 到 LLM 的流式增量提交。
- 不引入双轨 STT。
- 不把客户端本地 heuristic 作为默认正式打断逻辑。

## 3. 核心问题分析

### 3.1 当前问题不是单一阈值问题

现象上看像是“打断不灵敏”，但根因分成两条链路：

1. 停播链路  
   - 是否足够早地停止 TTS。

2. 识别链路  
   - TTS 停掉之后，用户开头那一小段音频是否还保留在本轮 utterance 中。

如果只调 detector 阈值，只能影响“停得早不早”，不能解决“停了以后为什么只剩后半句”。

### 3.2 为什么会丢前半句

当用户开口打断时，最容易丢失的是开头 `200~500ms` 的音频。原因通常是：

1. 用户刚开口时 TTS 还在播，前几帧混有回声与人声。
2. 服务端在确认打断前，这一小段音频可能已经被 STT 消耗，但尚未形成稳定 final。
3. 一旦切到新一轮，旧音频没有被明确保留或重新并入当前 utterance。

结果就是：

- TTS 停了，但前几个字已经过去了。
- STT 最后只留下后半句。

### 3.3 为什么不能靠“等待窗口”解决

等待 `300~800ms` 的稳定窗口，确实可以提高 final 完整度，但用户已明确：

- 不能接受额外等待来换整句保留。

因此本次设计必须在“不增加提交等待感知”的前提下解决问题。唯一可行方向是：

1. 更早停播
2. 保留停播前短时间的原始音频

## 4. 推荐方案

推荐采用：

**激进早停 + 预滚动音频缓冲 + final-only 提交**

### 4.1 方案概述

把打断拆成两件事：

1. 早停  
   - 一检测到明显开口迹象，就尽快停 TTS。
   - 目标是给用户语音让路，而不是立即提交给 LLM。

2. 补头  
   - 服务端始终保存最近一小段麦克风原始 PCM（预滚动缓冲，pre-roll）。
   - 一旦打断发生，把这段 pre-roll 音频并入当前用户 utterance。
   - 后续继续收实时音频，最终仍由 STT `final` 提交给 LLM。

### 4.2 为什么推荐这个方案

它同时满足四个约束：

1. 比当前更快停播
2. 不要求等待额外稳定窗口
3. 不引入 partial 直喂 LLM
4. 不需要重写整套识别系统

### 4.3 不采用的方案

#### 方案 B：仅继续调 detector 阈值

不推荐。  
原因：只能影响“停播时机”，不能解决“打断开头音频丢失”。

#### 方案 C：partial 直接流式喂 LLM

不推荐。  
原因：当前链路还未稳定，若在此时把 partial 直接送给 LLM，会引入：

- partial 改写
- 提前错误回复
- turn 状态更混乱

#### 方案 D：客户端本地优先停播

不推荐。  
原因：会导致 Web 与 Flutter 两端状态机不一致，增加维护和论文描述成本。

## 5. 状态机设计

建议将语音会话明确为以下状态：

### 5.1 `IDLE`

- 当前无 TTS 播放
- 正常收音
- 正常 STT 识别

### 5.2 `ASSISTANT_SPEAKING`

- 当前 TTS 正在播报
- 同时持续维护一段 `mic_pre_roll_buffer`
- 动态打断算法在此状态下持续工作

### 5.3 `INTERRUPT_STOPPING`

- 检测到用户明显开口
- 立刻执行停播
- 清理当前 assistant turn
- 将 pre-roll 准备并入用户 utterance

这是一个很短的过渡状态，职责只有两件：

1. 停 TTS
2. 补 pre-roll

### 5.4 `USER_TURN_COLLECTING`

- TTS 已停
- 当前正在收集用户这一轮语音
- `partial` 只展示
- `final` 才提交给 LLM

## 6. 数据流设计

### 6.1 正常播放期间

1. 客户端麦克风音频持续上传
2. 服务端每收到音频：
   - 写入 `mic_pre_roll_buffer`
   - 同时送入 detector / STT
3. 若无打断，assistant 正常播完

### 6.2 用户开口打断时

1. 在 `ASSISTANT_SPEAKING` 状态下出现“明显开口迹象”
2. 立即进入 `INTERRUPT_STOPPING`
3. 服务端执行：
   - `tts_reset`
   - `tts_interrupted`
   - 失效当前 turn generation
4. 从 `mic_pre_roll_buffer` 取出最近 `300~500ms` 音频
5. 将该段 pre-roll 并入当前用户 utterance 的开头
6. 切换到 `USER_TURN_COLLECTING`
7. 后续继续接收实时音频，直到 STT 产出 `final`
8. 用该 `final` 调用 LLM

### 6.3 提交策略

- `partial`：只用于显示与打断辅助，不提交 LLM
- `final`：唯一允许提交给 LLM 的文本

## 7. 模块改动设计

本次实现尽量只改 3 个核心位置。

### 7.1 `backend/api/voice_session.py`

这是主改动点。

新增内容：

1. `mic_pre_roll_buffer`
   - 保存最近 `300~500ms` 原始 PCM16 音频

2. `interrupt_mode` 或等价状态字段
   - 表示当前是否处于：
     - 正常播报
     - 正在停播
     - 收集用户 turn

3. `pending_user_audio_seeded`
   - 标识 pre-roll 是否已经并入当前 utterance

主要改动逻辑：

1. `_receive_loop()`
   - 每次收到麦克风音频，先写入 pre-roll buffer

2. `_handle_barge_in()`
   - 保留动态 detector
   - 但新增“早停入口”

3. `_interrupt_now()`
   - 负责尽快停播与 generation 失效

4. STT 喂音逻辑
   - 打断后不是只从“停之后”开始喂
   - 而是将 `pre-roll + 当前帧 + 后续实时帧` 作为本轮 utterance 输入

### 7.2 `backend/voice/barge_in_detector.py`

只做小改，不推翻现有算法。

建议新增一个工程事件：

- `EARLY_TRIGGER`

语义：

- 用户开口迹象已经足够强，允许先停播
- 但这不等价于论文里的“正式 confirmed 打断”

事件分工：

- `EARLY_TRIGGER`：用于尽快停 TTS
- `CONFIRMED`：继续保留为论文实验与正式 detector 结果

### 7.3 STT 接入层

尽量不改 provider 本体，先在 `voice_session.py` 侧完成。

若当前 STT 在打断切换时必须 reset，则顺序必须明确：

1. reset 当前识别状态
2. 立刻回灌 pre-roll
3. 再继续喂当前与后续实时音频

避免的问题：

- 开头几个字丢失
- 同一段音频重复提交

## 8. 参数建议

建议先以以下参数作为初始值：

- `pre_roll_ms = 400`
- `early_trigger_min_frames = 2`
- `early_trigger_energy_scale = 1.3 ~ 1.6 * noise_floor`
- `final_only_submit = true`

这些参数的目标不是“一次到位”，而是先把链路跑通，再做精调。

## 9. 风险与应对

### 9.1 误打断增加

由于用户选择“更快停播优先”，误打断会比当前更容易发生。  
这是本方案的明确 trade-off。

应对方式：

- 保留现有 detector 评分与日志
- 后续通过 `EARLY_TRIGGER` 与 `CONFIRMED` 双事件分层观测误打断来源

### 9.2 pre-roll 带入少量 TTS 回声

由于 pre-roll 本来发生在 TTS 尚未完全停掉前，带入少量回声是可能的。

应对方式：

- 先接受这一代价，因为它比“开头直接丢失”更可控
- 后续可再用回声抑制或更窄的 pre-roll 时长优化

### 9.3 STT reset 不当导致重复识别

应对方式：

- 明确 reset、补头、继续送音的固定顺序
- 增加去重保护，避免同一 final 连续提交

## 10. 测试设计

### 10.1 停播测试

目标：

- 用户一开口，TTS 能尽快停

观察事件：

- `tts_start`
- `barge_in`
- `tts_reset`
- `tts_interrupted`
- `tts_done`

关注点：

- 打断发生时，是否在真实播放期间出现 `tts_interrupted`

### 10.2 整句保留测试

固定测试句：

1. `你重新说一遍刚刚那句话`
2. `等一下你先别说了`
3. `你叫什么名字`
4. `你在干嘛`
5. `为什么不喊我宝贝`

目标：

- final 文本尽量保住完整句子
- 不再明显只剩后半句

### 10.3 误打断测试

场景：

- 只播放 TTS，不说话

目标：

- 误打断次数可控

### 10.4 回归测试

确认以下能力不受影响：

- greeting 播放
- 普通对话识别
- `final -> LLM -> TTS` 正常闭环

## 11. 实施顺序

建议按以下顺序落地：

1. 在 `voice_session.py` 增加 `mic_pre_roll_buffer`
2. 打通 `INTERRUPT_STOPPING -> USER_TURN_COLLECTING`
3. 接入 pre-roll 回灌逻辑
4. 在 detector 中增加 `EARLY_TRIGGER`
5. 加事件日志与去重保护
6. 做真机回归测试

## 12. 结论

本次调优不再继续围绕“单纯调 detector 阈值”展开，而是通过：

1. 更激进地停播
2. 保留停播前短时间原始音频
3. 继续坚持 `final-only` 提交

来解决“打断够快，但前半句经常丢失”的核心问题。

这是在当前项目约束下，最符合用户优先级、改动范围最可控、也最容易形成论文描述的一条方案。
