# 自适应打断检测算法设计与实现文档

## 1. 文档定位

本文档服务于两个目标：

1. 指导后续代码开发、模块重构、参数调优和测试设计
2. 为毕业论文中“问题分析、算法设计、系统实现、实验评估”章节提供可直接复用的技术材料

当前项目的完整语音链路为：

`客户端录音 -> WebSocket 上传 -> 服务端 STT -> RAG/LLM -> TTS -> WebSocket 回传播放`

打断检测位于服务端会话运行时，是整个实时交互体验的关键环节。当前工程中的实现已经具备可运行基础，核心代码位于 `backend/api/voice_session.py`。后续建议将其正式命名为“自适应打断检测算法”。

---

## 2. 问题背景与研究动机

在厨房语音助手场景中，系统通常需要一边播报，一边允许用户随时插话。例如：

- “下一步是什么？”
- “等一下，盐放多少？”
- “停一下，我没听清”

这一场景和普通的单向语音播放不同，存在以下难点：

1. 厨房背景噪声持续存在，且强度变化明显
   - 油烟机
   - 排风扇
   - 水流声
   - 锅铲碰撞
   - 沸腾和爆炒声

2. TTS 通过扬声器播放时，回放音频会再次进入麦克风形成回声

3. 用户插话往往是短句且突发的
   - 既要求快速停止 TTS
   - 又要避免将回声和环境噪声误判为用户打断

如果直接使用固定能量阈值或简单 VAD，会出现两个典型问题：

1. 在噪声增强时漏检真实打断
2. 在 TTS 刚播放完或播放期间误将回声判成打断

因此，需要一种不是依赖固定门限，而是能根据环境噪声、语音连续性和回声风险动态调整判断条件的实时检测方法。

---

## 3. 算法目标与设计原则

### 3.1 目标

本算法的目标是：

1. 在系统播报期间实时检测用户是否有打断意图
2. 对“疑似打断”和“确认打断”分层处理
3. 抑制回声、背景噪声和短促干扰造成的误触发
4. 尽量缩短从用户开口到 TTS 停止的时间
5. 保持算法可解释、可调参、可测试、可论文复现

### 3.2 设计原则

1. 使用轻量实时算法，避免引入高延迟深度模型
2. 不只看单帧，而是结合时间连续性
3. 不使用固定阈值，而使用自适应噪声基线
4. 不将 VAD 输出直接等同于打断判定，而是与其他特征融合
5. 显式建模 TTS 回声风险，避免“自己打断自己”

---

## 4. 算法定义

### 4.1 输入

算法每次接收一段来自客户端的原始音频流，具体为：

- 采样率：`16kHz`
- 声道数：`1`
- 编码格式：`PCM16`
- 帧长：`20ms`

此外，还依赖会话上下文信息：

- 当前 TTS 是否正在播报
- 最近一次 TTS 音频输出的时间戳
- 最近一次 TTS 结束的时间戳

### 4.2 输出

算法输出不是简单的二分类，而是状态化结果：

- `no_interrupt`
- `interrupt_pending`
- `interrupt_confirmed`
- `resume_speaking`

在工程实现中，这些状态被编码为：

- `IDLE`
- `SPEAKING`
- `INTERRUPT_PENDING`
- `INTERRUPTED`

### 4.3 与普通 VAD 的区别

普通 VAD 只判断“当前帧是否像人声”，并不回答以下问题：

- 这段人声是否发生在系统播报期间
- 它是否足够连续，足以构成用户打断
- 它是否只是 TTS 回声

因此，VAD 只能作为打断检测的一个子特征，而不能直接替代打断检测算法。

## 5. 整体流程

算法在 `VoiceSessionRunner._handle_barge_in()` 中以逐帧方式工作，可概括为 7 个步骤：

1. 缓冲输入音频并按固定长度切分为 20ms 帧
2. 对每帧进行 VAD 或能量回退判定
3. 计算当前帧短时能量
4. 更新最近窗口中的语音比例
5. 根据最近 TTS 时间计算回声风险
6. 计算综合打断评分
7. 驱动状态机决定是否进入疑似打断、确认打断或恢复播报

简化流程如下：

```text
音频帧
  -> VAD/能量判定
  -> 特征提取
  -> 综合评分
  -> 状态机判定
  -> 暂停/停止 TTS 或恢复播报
```

## 6. 特征设计

本算法采用“多特征融合”的思路。当前实现中的主要特征如下。

### 6.1 `is_speech`

含义：

- 当前帧是否被判断为语音帧

来源：

- 优先使用 `webrtcvad`
- 若环境缺少 `webrtcvad`，则使用能量门限回退方法

意义：

- 提供基础的语音活动线索
- 但不单独作为打断依据

### 6.2 `energy`

含义：

- 当前帧的平均绝对能量

实现：

- 将 PCM16 音频转为 `int16`
- 计算样本绝对值均值

意义：

- 反映当前音频的强弱变化
- 有助于识别用户突然开口时的能量抬升

### 6.3 `noise_floor`

含义：

- 环境噪声底线

本质：

- 它不是常数，而是一个动态更新的背景能量估计值

作用：

- 用于将当前能量与背景环境进行相对比较
- 避免在安静环境和嘈杂环境中使用相同阈值

### 6.4 `speech_ratio`

含义：

- 最近一段滑动窗口内语音帧所占比例

意义：

- 单独一帧像语音，不代表用户真的开始说话
- 如果最近若干帧持续被判为语音，则更可能是用户插话

### 6.5 `echo_risk`

含义：

- 当前帧处于 TTS 回声敏感区的风险程度

来源：

- 根据当前时间与最近一次 TTS 输出时间的差值估算

意义：

- 越接近 TTS 输出时刻，越可能采到扬声器回放而非真实打断
- 这一特征用于抑制“系统把自己的声音当成用户说话”

## 7. 数学建模与评分公式

### 7.1 动态噪声基线更新

设：

- `E_t` 表示第 `t` 帧能量
- `N_t` 表示第 `t` 帧噪声底线
- `alpha` 表示更新速率

则噪声底线可写为：

```text
N_t = (1 - alpha) * N_(t-1) + alpha * E_t
```

为避免在高语音置信区间内噪声底线过快漂移，当前工程实现中：

- 若当前帧判定为语音，则降低更新速率
- 若当前系统处于 `SPEAKING` 状态，也进一步降低更新速率

这意味着噪声基线对“真实环境变化”敏感，但对“短暂说话能量上升”不过度跟随。

### 7.2 能量增益

为了用相对值而非绝对值描述当前帧是否高于背景，定义能量增益：

```text
G_t = max(0, (E_t - N_t) / N_t)
```

意义：

- 若当前帧能量仅与噪声基线相当，则增益接近 0
- 若用户明显开口，增益上升

### 7.3 综合评分

当前工程实现中，综合评分可概括为：

```text
S_t = V_t + A_t + R_t - H_t
```

其中：

- `V_t`：VAD 评分
- `A_t`：能量增益评分
- `R_t`：连续语音比例评分
- `H_t`：回声惩罚项

可展开为：

```text
vad_score = 0.8 if is_speech else 0
energy_score = min(1.5, G_t / adaptive_energy_margin)
ratio_score = min(1.0, speech_ratio * 1.2)
echo_penalty = echo_risk * 0.9

score = vad_score + energy_score + ratio_score - echo_penalty
```

### 7.4 评分解释

1. `vad_score`
   - 当前帧若像语音，则给基础分

2. `energy_score`
   - 当前帧若明显高于背景噪声，则加分

3. `ratio_score`
   - 若最近一段时间持续像语音，则加分

4. `echo_penalty`
   - 若距离最近 TTS 输出太近，则减分

这种设计的优点是：

1. 简单可解释
2. 实时性强
3. 易于做参数搜索和消融实验

## 8. 状态机设计

### 8.1 状态定义

系统采用四状态模型：

1. `IDLE`
   - 当前未播报 TTS

2. `SPEAKING`
   - 当前正在播放 TTS

3. `INTERRUPT_PENDING`
   - 已怀疑用户开始打断
   - 暂停向客户端继续推送 TTS 音频
   - 但还未正式确认打断

4. `INTERRUPTED`
   - 已确认打断
   - 立即停止当前 TTS

### 8.2 状态转移逻辑

#### `IDLE -> SPEAKING`

触发条件：

- TTS 开始播放

#### `SPEAKING -> INTERRUPT_PENDING`

触发条件：

- 当前为语音帧
- 连续语音帧达到起始门限
- 综合评分超过 `pending_score`

对应行为：

- 临时停止继续发送 TTS 音频
- 向客户端发送 `tts_reset` 准备停播

#### `INTERRUPT_PENDING -> INTERRUPTED`

触发条件：

- 连续语音帧达到确认门限
- 综合评分超过 `confirm_score`

对应行为：

- 调用 `tts.stop()`
- 发送 `barge_in` 与 `tts_interrupted` 事件

#### `INTERRUPT_PENDING -> SPEAKING`

触发条件：

- 进入 `pending` 后，后续没有持续语音
- 静默帧达到回退门限

对应行为：

- 视为误警报
- 恢复播报

#### `SPEAKING/INTERRUPT_PENDING/INTERRUPTED -> IDLE`

触发条件：

- TTS 正常结束或手动停止

### 8.3 为什么要使用 `pending`

如果检测到一帧疑似语音就立刻停播，系统在厨房环境中会极不稳定。加入 `pending` 的意义在于：

1. 为真实插话提供快速过渡
2. 为回声或瞬时噪声提供缓冲区
3. 平衡“打断速度”和“误触发率”

## 9. 动态确认策略

### 9.1 问题

若始终固定要求用户连续说话 800ms 或 1000ms 才确认打断，则会出现：

1. 短句打断太慢
2. 真实打断体验变差
3. 高置信场景下反应不够快

### 9.2 当前策略

当前工程采用“分数越高，确认所需帧数越少”的策略。

设：

- `base_frames = min_speech_ms / frame_ms`

则：

```text
if score >= confirm_score + 0.75:
    confirm_frames = base_frames / 3
elif score >= confirm_score + 0.35:
    confirm_frames = base_frames / 2
else:
    confirm_frames = base_frames
```

### 9.3 优点

1. 当用户开口很明显时，系统更快响应
2. 当判定边缘时，系统保持谨慎
3. 算法既有自适应性，又不复杂

这部分非常适合写入论文，作为“动态确认时长”的一个具体创新点。

## 12. 开发建议

### 12.1 建议的模块重构

当前逻辑集中在 `VoiceSessionRunner` 中，便于联调，但不利于算法独立演进。后续建议：

1. 新建 `backend/voice/barge_in_detector.py`
2. 将以下能力拆出为独立类：
   - 特征提取
   - 噪声基线维护
   - 评分器
   - 状态机
3. `VoiceSessionRunner` 只负责：
   - 喂入音频
   - 接收检测结果
   - 控制 TTS

这样做的好处：

1. 更便于写单元测试
2. 更便于做离线评估复用
3. 更适合在论文中描述清晰的模块边界

### 12.2 建议新增的调试信息

后续建议在测试模式下增加每帧日志或结构化输出，至少记录：

- `frame_index`
- `energy`
- `noise_floor`
- `is_speech`
- `speech_ratio`
- `echo_risk`
- `score`
- `state_before`
- `state_after`

这样可以直接画出：

- 分数随时间变化曲线
- 不同噪声场景下的阈值命中情况
- 用户打断前后的状态机转移图

### 12.3 建议新增的测试覆盖

1. 单元测试
   - 回声风险升高时评分下降
   - 高能量持续输入时进入 `pending`
   - 高置信持续输入时进入 `interrupted`
   - 短暂误触发后能恢复 `speaking`

2. 集成测试
   - 模拟 TTS 播放 + 人声插话
   - 模拟 TTS 回声 + 无真实插话
   - 模拟厨房突发噪声 + 无真实插话

3. 离线评估
   - 基于真实采样数据跑批量统计

## 13. 实验设计建议

### 13.1 数据集构成

建议建立面向厨房场景的小规模专用数据集，而不是直接套用通用 VAD 数据集。

样本建议分为三类：

1. 正样本
   - TTS 播放期间，用户真实打断

2. 负样本 A
   - TTS 播放期间，无用户说话，仅有回声

3. 负样本 B
   - TTS 播放期间，无用户说话，仅有厨房噪声或突发声

建议覆盖的环境：

- 安静室内
- 风扇声
- 水流声
- 油烟机声
- 锅铲碰撞声
- 不同 TTS 音量
- 不同说话距离

### 13.2 标注格式

当前项目中的 `scripts/eval_barge_in.py` 已支持 JSONL 清单。建议统一采用如下格式：

```json
{"path":"samples/interrupt_01.wav","label":"interrupt","expected_interrupt_ms":820,"tts_offset_ms":0}
{"path":"samples/echo_only_01.wav","label":"no_interrupt","tts_offset_ms":0}
```

字段说明：

- `path`
  - 样本音频路径

- `label`
  - `interrupt` 或 `no_interrupt`

- `expected_interrupt_ms`
  - 用户真实开始插话的时间点

- `tts_offset_ms`
  - 样本起点相对于最近一次 TTS 输出的偏移

### 13.3 推荐指标

至少统计以下指标：

1. 误打断率 `False Interrupt Rate`
2. 漏检率 `Miss Rate`
3. 精确率 `Precision`
4. 召回率 `Recall`
5. `F1`
6. 平均检测时延 `Average Detect Latency`
7. 时延偏差 `Latency Delta`

若条件允许，可增加：

1. P95 检测时延
2. 用户开口到 TTS 实际停止的端到端时延

### 13.4 对比实验建议

建议至少做三组：

1. Baseline 1
   - 固定能量阈值 + 固定确认时长

2. Baseline 2
   - VAD 连续帧计数法

3. Proposed
   - 动态噪声基线 + 综合评分 + 动态确认时长

### 13.5 消融实验建议

建议做以下消融：

1. 去掉 `echo_risk`
2. 去掉 `speech_ratio`
3. 去掉动态 `noise_floor`
4. 去掉动态确认帧数

这样可以证明每一项设计对最终效果都有贡献。

## 15. 当前阶段结论

对于本项目，自适应打断检测算法是最适合作为毕设核心技术点之一的模块，原因如下：

1. 模块解耦与测试增强
2. 数据采集与离线评估
3. 参数对比与论文实验表格整理
