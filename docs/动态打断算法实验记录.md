## 3. 初始问题

初始离线评估采用较简单的打断判定方式：

- 动态噪声底线
- 连续语音比例
- 综合分数阈值

初始结果如下：

| 指标 | 数值 |
| --- | --- |
| TP | 6 |
| FN | 0 |
| FP | 6 |
| TN | 0 |
| Precision | 0.50 |
| Recall | 1.00 |

问题非常明显：

- 所有正样本都能被触发
- 所有负样本也都被误判

这说明原始策略更像是在检测“强声音事件”，而不是稳定地区分“用户插话”与“机器回声/环境噪声”。

## 4. 样本特征分析

基于 `samples/outputs/barge_feature_stats.json` 可以得到几个关键现象。

### 4.1 正样本特征

正样本普遍具有以下特点：

- `max_energy` 明显更高
- `max_to_base_ratio` 较高，通常在 `13~24`
- `frames_ge_4000` 一般大于等于 `3`
- `longest_run_ge_4000` 通常在 `3~7`
- 强峰值大多出现在播报开始后 `2 秒` 内

### 4.2 普通负样本特征

大部分 `echo_only` 与普通 `noise_only` 负样本具有以下特点：

- `max_energy` 明显低于正样本
- `frames_ge_4000 = 0`
- `max_to_base_ratio` 较低
- 不具备足够强的突发峰值

### 4.3 硬负样本特征

`samples/barge_in/barge_noise_only_02.wav` 是一条典型硬负样本，其特征如下：

- `max_energy = 13281`
- `frames_ge_4000 = 29`
- `longest_run_ge_4000 = 24`
- `first_ge_2000_ms = 2200`

这说明该样本虽然不是用户插话，但在能量和峰值上非常接近甚至超过部分正样本，仅靠普通能量门限很难区分。

## 5. 动态打断算法改造

针对上述问题，打断算法在原有基础上增加了三类规则。

### 5.1 动态峰值门限

新增参数：

- `peak_gate_min`
- `peak_gate_scale`
- `peak_min_frames`

核心思想：

- 不能只看平均能量
- 必须出现足够强的高能量峰值
- 并且高峰值至少持续若干帧

判定门限形式：

```text
peak_gate = max(peak_gate_min, noise_floor * peak_gate_scale)
```

只有当语音片段中满足：

- `energy >= peak_gate`
- 且高峰值帧数达到 `peak_min_frames`

才允许进入最终打断确认。

### 5.2 平台抑制

新增参数：

- `peak_plateau_limit`

核心思想：

- 用户插话通常表现为短促而明显的峰值簇
- 连续时间过长的高能量平台更像稳定噪声或设备声

因此加入：

```text
max_peak_run <= peak_plateau_limit
```

用于抑制长时间持续高能量的平台型噪声。

### 5.3 晚期打断更严格

新增参数：

- `late_window_ms`
- `late_peak_min_frames`
- `late_plateau_limit`

核心思想：

- 用户主动打断通常发生在 TTS 播报前半段
- 当疑似打断出现得较晚时，应提高确认门槛

因此在 `pending_tts_elapsed_ms > late_window_ms` 时，算法采用更严格的规则：

- 提高所需高峰值帧数
- 收紧允许的平台持续长度

这一步专门用于处理像 `barge_noise_only_02.wav` 这种“晚出现、但峰值很高”的硬负样本。

- 但足以说明动态规则对当前手机外录场景具有明显改进作用

## 8. 可以直接写进论文的结论

建议使用以下表述：

1. 初始打断算法在真实外录样本上能够检测到用户插话，但对回声和环境噪声敏感，误打断较多。
2. 基于样本特征统计发现，真实用户打断通常伴随更高的瞬时峰值、更明显的相对能量提升，以及更短促的高能量峰值簇。
3. 因此本文在原始多特征评分基础上引入动态峰值门限、平台抑制和晚期打断增强判定机制。
4. 在自建 12 条小规模样本集上，改进后的动态打断算法相较初始方案显著降低了误打断。

## 9. 论文中的风险表述

为了避免答辩时被追问过度结论，建议同时写明：

- 样本数量较少
- 当前实验主要用于验证方案可行性
- 后续仍需在更多设备、更多噪声场景下继续验证
