# 自适应打断检测实验设计与论文数据指南

## 1. 文档目的

本文档用于解决三个实际问题：

1. 后续如何采集打断样本并形成可复用数据集
2. 如何评估自适应打断算法是否真的优于简单阈值方法
3. 如何把实验数据整理成论文中的表格、曲线和结论

本文档与 `docs/adaptive-barge-in-design.md` 配套使用：

- `adaptive-barge-in-design.md` 负责解释算法本身
- 本文档负责解释“怎么测、测什么、结果怎么写”

---

## 2. 实验目标

自适应打断检测算法的实验目标不是单纯“能不能触发打断”，而是回答以下问题：

1. 在真实厨房噪声条件下，系统能否稳定识别用户插话
2. 算法是否能抑制 TTS 回声导致的误打断
3. 与固定阈值方案相比，是否在误打断率和响应速度之间取得更好平衡
4. 不同参数组合、不同噪声场景下，算法表现如何变化

---

## 3. 评估思路总览

建议采用“两层实验”：

1. 离线实验
   - 使用标注好的录音样本批量评估
   - 输出精确率、召回率、误打断率、平均检测时延等客观指标

2. 在线联调实验
   - 通过 `frontend/index.html` 实时调参
   - 观察 `barge_in` 事件和统计结果
   - 用于工程联调和参数搜索

两者关系如下：

- 离线实验负责论文中的主结果
- 在线实验负责开发阶段快速定位参数问题

---

## 4. 数据集设计

### 4.1 数据来源原则

建议自己录制样本，不建议完全依赖公开数据集。原因是：

1. 你的系统场景是“厨房 + TTS 播报 + 用户插话”
2. 通用语音数据集通常不包含真实扬声器回声和厨房场景噪声
3. 你的论文更需要“场景贴合”，而不是“数据规模特别大”

### 4.2 样本类别

建议至少采集以下三类样本。

#### A. 正样本：真实打断

定义：

- 系统正在播放 TTS
- 用户在播报中途开口打断

示例：

- “等一下”
- “停”
- “下一步”
- “盐放多少”

#### B. 负样本：回声但无人打断

定义：

- 系统正在播放 TTS
- 麦克风采到扬声器回放
- 但用户没有说话

这是验证 `echo_risk` 是否有效的关键样本。

#### C. 负样本：厨房噪声但无人打断

定义：

- 系统正在播放 TTS
- 存在风扇、水流、锅铲等背景声
- 用户没有说话

这类样本用来测试算法对环境噪声的鲁棒性。

### 4.3 场景维度建议

每类样本建议再按环境分组：

1. 安静环境
2. 风扇声
3. 水流声
4. 油烟机声
5. 锅铲碰撞声
6. 高音量 TTS
7. 低音量 TTS
8. 近距离说话
9. 中距离说话

### 4.4 数据规模建议

本科毕设不要求大数据规模，但也不能过少。建议最低配置如下：

1. 正样本：`50` 条以上
2. 负样本（回声）：`30` 条以上
3. 负样本（厨房噪声）：`30` 条以上

更稳妥的建议：

1. 总样本数达到 `150 ~ 300` 条
2. 每种噪声场景至少 `20` 条样本

## 5. 数据采集方法

### 5.1 采集方式

建议使用与你当前系统一致的采集链路：

1. 客户端播放 TTS
2. 同时用麦克风录制环境输入
3. 保留 TTS 回放泄漏到麦克风的真实效果

不要直接用“纯净人声 + 后期混音”完全替代真实录音，因为那样难以真实反映设备回声路径。

### 5.2 录音要求

统一录制为：

- `wav`
- `16kHz`
- 单声道
- `16bit PCM`

每条样本建议时长：

- `3 ~ 8` 秒

### 5.3 录音内容建议

正样本建议覆盖三种打断形式：

1. 短命令
   - “停”
   - “等一下”

2. 问答型打断
   - “盐放多少”
   - “要不要加水”

3. 连续句式打断
   - “等一下我还没切完”

## 6. 样本标注规范

### 6.1 标注目标

对于正样本，需要标出“用户真实开始插话”的时间点。这个时间点用于计算检测时延。

### 6.2 推荐标注工具

可使用：

1. Audacity
2. Adobe Audition
3. Praat

只要能看到波形并支持毫秒级定位即可。

### 6.3 JSONL 标注格式

当前仓库的 `scripts/eval_barge_in.py` 已支持 JSONL 格式，建议统一如下：

```json
{"path":"samples/interrupt_01.wav","label":"interrupt","expected_interrupt_ms":820,"tts_offset_ms":0}
{"path":"samples/echo_only_01.wav","label":"no_interrupt","tts_offset_ms":0}
```

字段说明：

- `path`
  - 样本文件路径

- `label`
  - `interrupt` 表示正样本
  - `no_interrupt` 表示负样本

- `expected_interrupt_ms`
  - 用户真实开始打断的时间点，仅正样本必填

- `tts_offset_ms`
  - 样本起始相对于最近一次 TTS 输出的时间偏移
  - 用于模拟回声敏感区

- `config`
  - 可选
  - 用于针对某些样本覆盖默认参数

### 6.4 标注原则

1. 以“用户真实开始发声”的起点为准
2. 若是短促语气词，以第一处明显声学起点为准
3. 对负样本不需要填写 `expected_interrupt_ms`
4. 保持不同标注者标准一致

## 7. 实验工具链

### 7.1 离线评估脚本

主工具：

- `scripts/eval_barge_in.py`

输入：

- JSONL 样本清单

输出：

- 每条样本的预测结果
- 汇总统计结果

当前脚本已经支持：

1. 判断是否预测为打断
2. 输出首次进入 `pending` 的时间
3. 输出首次确认 `interrupted` 的时间
4. 统计精确率、召回率、误打断率、漏检率

前端测试页：

- `frontend/index.html`

支持能力：

1. 实时修改打断参数
2. 拉取当前打断状态
3. 导出 `barge_in` JSON 和 CSV

在线联调适合：

1. 快速调参数
2. 观察事件是否合理
3. 找问题但不作为最终论文主实验

## 8. 评估指标

### 8.1 分类指标

设：

- `TP`：正样本中被正确检测为打断
- `FN`：正样本中漏检
- `FP`：负样本中误判为打断
- `TN`：负样本中正确保持未打断

推荐指标：

1. 精确率 `Precision = TP / (TP + FP)`
2. 召回率 `Recall = TP / (TP + FN)`
3. `F1`
4. 误打断率 `False Interrupt Rate = FP / Negative Samples`
5. 漏检率 `Miss Rate = FN / Positive Samples`

### 8.2 时延指标

对于正样本，还应计算：

1. 检测时延
   - `first_interrupted_ms - expected_interrupt_ms`

2. 平均检测时延

3. P95 检测时延

若有条件，也可增加：

1. 用户开口到 TTS 实际停播的端到端时延

### 8.3 辅助分析指标

从在线或调试日志中可统计：

1. 平均 `score`
2. 平均 `echo_risk`
3. 平均 `speech_ratio`
4. 平均 `noise_floor`

这些指标更适合做参数分析和消融解释，而不是主结论。

## 9. Baseline 与对比实验设计

为了证明你的算法有效，建议至少设计两个基线。

### 9.1 Baseline 1：固定能量阈值法

规则：

1. 当前帧能量超过固定阈值
2. 连续若干帧满足条件
3. 即判定打断

问题：

1. 不适应噪声变化
2. 易受回声影响

### 9.2 Baseline 2：VAD 连续帧法

规则：

1. 当前帧由 VAD 判定为语音
2. 连续若干帧均为语音
3. 即判定打断

问题：

1. 忽略能量相对变化
2. 忽略回声时序信息

### 9.3 Proposed：自适应打断算法

规则：

1. 动态噪声基线
2. 综合评分
3. 回声惩罚
4. 双阈值状态机
5. 动态确认时长

这样论文中就能清楚展示“为什么 proposed 优于简单规则法”。

---

## 10. 消融实验设计

建议至少做以下消融实验。

### 10.1 去掉 `echo_risk`

目的：

- 验证回声风险建模是否真的降低了误打断率

预期现象：

- 在回声样本上误打断显著增加

### 10.2 去掉 `speech_ratio`

目的：

- 验证连续语音比例对稳定性的贡献

预期现象：

- 对短促噪声更敏感

### 10.3 去掉动态 `noise_floor`

目的：

- 验证自适应噪声估计的作用

预期现象：

- 在不同噪声环境中性能波动变大

### 10.4 去掉动态确认时长

目的：

- 验证高置信场景下的快速响应机制是否有效

预期现象：

- 检测时延上升

---

## 11. 参数搜索实验

### 11.1 推荐调参顺序

建议先固定算法结构，再调以下参数：

1. `pending_score`
2. `confirm_score`
3. `min_speech_ms`
4. `echo_suppression_window_ms`
5. `adaptive_energy_margin`
6. `noise_floor_alpha`

### 11.2 调参原则

若出现以下现象，可按如下方式调整。

#### 误触发多

表现：

- 没人打断也被判定中断

可尝试：

1. 提高 `pending_score`
2. 提高 `confirm_score`
3. 增大 `echo_suppression_window_ms`
4. 提高 `adaptive_energy_margin`

#### 打断反应慢

表现：

- 用户已经说话，但系统停播太慢

可尝试：

1. 降低 `confirm_score`
2. 降低 `min_speech_ms`
3. 适度降低 `pending_score`

#### 噪声环境下漏检

表现：

- 用户确实开口，但系统不响应

可尝试：

1. 适度降低 `adaptive_energy_margin`
2. 适度增加 `noise_floor_alpha`
3. 检查 `speech_ratio_window` 是否过大

---

## 12. 推荐实验流程

### 12.1 离线实验流程

1. 录制或整理样本音频
2. 用 Audacity 标注 `expected_interrupt_ms`
3. 生成 JSONL 清单
4. 运行：

```bash
python scripts/eval_barge_in.py path/to/manifest.jsonl --output-json eval_result.json
```

5. 统计每组参数的：
   - `TP`
   - `FN`
   - `FP`
   - `TN`
   - 平均检测时延
6. 做对比表与曲线图

### 12.2 在线实验流程

1. 启动后端和测试页
2. 在测试页填写一组参数
3. 执行真实交互
4. 导出 `barge_in` JSON/CSV
5. 观察 `score / echo_risk / speech_ratio`
6. 修改参数重复实验

在线实验主要用来辅助理解参数变化，不建议替代离线主实验。

## 15. 论文结果分析写法模板

你后续可以直接按下面思路写结果分析。

### 15.1 主实验结论模板

在厨房场景打断样本集上，与固定能量阈值方法和 VAD 连续帧方法相比，本文提出的自适应打断检测算法在误打断率和漏检率之间取得了更优平衡。其原因在于该方法不仅考虑当前帧是否为语音，还结合了动态噪声基线、连续语音比例和回声风险，从而提升了复杂环境下的稳定性。

### 15.2 回声抑制贡献分析模板

消融实验表明，去除回声风险项后，系统在仅存在 TTS 回放而无人插话的负样本上误打断率显著上升。这说明在播报式语音助手中，仅依赖语音活动检测不足以有效区分真实用户开口与扬声器回声，引入回声风险建模是必要的。

### 15.3 动态确认时长分析模板

实验结果显示，动态确认时长机制能够在高置信插话场景下有效缩短系统响应时间，同时在边缘样本中维持较低误判率。相比固定确认时长，本文方法在实时性与稳定性之间取得了更合理的折中。
