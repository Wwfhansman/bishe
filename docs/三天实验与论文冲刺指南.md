# 三天实验与论文冲刺指南

## 1. 文档目的

本文档用于指导项目在仅剩 3 天、且需要兼顾上班的情况下，如何以最小工作量完成：

- 样本采集
- 实验执行
- 数据整理
- 论文实验章节写作

本文档默认采用“最小可交付方案”，目标不是做大规模研究，而是完成一套能够支撑毕设答辩的**小规模可行性验证**。

## 2. 范围冻结

这三天内，项目范围必须冻结为以下两条主线：

1. 自适应打断检测
2. 轻量噪声抑制与语音增强

以下内容不再继续扩展：

- 声纹识别
- 新的模型接入
- 大规模数据集实验
- 新前端功能
- 大范围代码重构

原因很直接：现在最重要的是形成“可演示、可测量、可写进论文”的闭环，而不是继续增加功能。

## 3. 总体策略

三天内的正确做法是：

1. 只录一批最小样本
2. 同一批样本重复用于多个实验
3. 通过现有脚本快速输出结果
4. 把结果整理成表格和论文描述

换句话说，要追求的是：

- 少量样本
- 完整链路
- 可复现结果
- 能写论文

而不是：

- 样本很多
- 实验很复杂
- 指标很全

### 4.7 每份样本具体怎么录

如果你不熟悉录音，直接按下面的方法做，不要自行发挥。

#### 4.7.1 统一录音要求

- 格式优先使用 `wav`
- 单声道
- 采样率尽量用 `16kHz`
- 安静和噪声场景都尽量使用同一台手机或同一套设备录制
- 说话音量保持正常，不要刻意喊叫
- 麦克风距离嘴部大约 `20~40cm`
- 每条音频开头和结尾保留约 `0.5~1 秒` 空白

如果手机录音默认不是 `16kHz wav`，也可以先录，后面再统一转换，不要因为格式问题卡住。

#### 4.7.2 STT 样本怎么录

STT 样本每条只说一句话，整条音频建议控制在 `3~5 秒`。

每条录制步骤：

1. 点击开始录音
2. 停顿约 `0.5 秒`
3. 正常语速说目标句子
4. 说完后停顿约 `1 秒`
5. 停止录音

不要做的事：

- 不要一句话说两遍
- 不要边笑边说
- 不要夹杂多余口头语
- 不要在一句里改词

6 条样本建议这样录：

1. `stt_quiet_01.wav`
   - 环境：安静
   - 内容：下一步是什么

2. `stt_quiet_02.wav`
   - 环境：安静
   - 内容：盐要放多少

3. `stt_fan_01.wav`
   - 环境：开风扇或油烟机
   - 内容：现在需要加水吗

4. `stt_fan_02.wav`
   - 环境：开风扇或油烟机
   - 内容：请重复上一句

5. `stt_water_01.wav`
   - 环境：开水流或厨房背景声
   - 内容：火候需要调小吗

6. `stt_water_02.wav`
   - 环境：开水流或厨房背景声
   - 内容：这个菜还要煮多久

#### 4.7.3 打断正样本怎么录

正样本的目标是：系统正在播报时，你开口打断。

每条音频建议控制在 `4~8 秒`。

最简单录法有两种，任选其一。

方法 A：直接录真实交互

1. 打开系统，让 TTS 正常播报
2. 让手机或录音设备同时录下整个过程
3. 等 TTS 播放约 `1~2 秒` 后，你说出打断词
4. 录到 TTS 停止或明显被打断后结束

方法 B：先外放 TTS，再人工插话

1. 准备一段系统播报音频或让系统现场播报
2. 用扬声器外放
3. 录音设备放在平时使用位置
4. 在播报中间插一句打断词

正样本建议内容如下：

1. `barge_interrupt_01.wav`
   - 打断词：等一下
   - 建议在播报开始后约 `1.2 秒` 插话

2. `barge_interrupt_02.wav`
   - 打断词：停
   - 建议在播报开始后约 `0.9 秒` 插话

3. `barge_interrupt_03.wav`
   - 打断词：重新说一遍
   - 建议在播报开始后约 `1.5 秒` 插话

4. `barge_interrupt_04.wav`
   - 打断词：下一步
   - 建议在播报开始后约 `1.1 秒` 插话

5. `barge_interrupt_05.wav`
   - 打断词：等等
   - 建议在播报开始后约 `1.3 秒` 插话

6. `barge_interrupt_06.wav`
   - 打断词：我没听清
   - 建议在播报开始后约 `1.0 秒` 插话

注意：

- 不要求你精确到毫秒
- 只要大致在 `1 秒左右` 到 `1.5 秒左右` 开口即可
- 后面再根据波形粗略标注 `expected_interrupt_ms`

#### 4.7.4 打断负样本怎么录

负样本分两种。

第一种：`echo_only`

目的：

- 只有系统播报和回声，没有人打断

录法：

1. 让系统正常 TTS 播报
2. 设备录下整个播报片段
3. 全程不要说话
4. 长度保持 `4~6 秒`

对应文件：

- `barge_echo_only_01.wav`
- `barge_echo_only_02.wav`
- `barge_echo_only_03.wav`

第二种：`noise_only`

目的：

- 有厨房背景噪声，但没有人打断

录法：

1. 保持系统 TTS 播报
2. 打开风扇、水流或其他厨房背景声
3. 全程不要说话
4. 录下 `4~6 秒`

对应文件：

- `barge_noise_only_01.wav`
- `barge_noise_only_02.wav`
- `barge_noise_only_03.wav`

#### 4.7.5 标注 `expected_interrupt_ms` 最简单的方法

如果你不会精确标注，就按下面最低要求做：

1. 记住你大概是在 TTS 开始后多久说话
2. 直接写一个近似值，例如：
   - `900`
   - `1100`
   - `1300`
3. 后面如果有时间，再用 Audacity 微调

对当前毕设来说，近似值是可以接受的，不要在标注上浪费太多时间。

## 6. 只做三组实验

为了保证三天内完成，实验只保留以下三组。

### 6.1 实验一：降噪前后对比

目的：

- 验证噪声抑制模块确实对输入语音产生了可观察影响

方法：

- 选取 3 到 6 条语音
- 输出增强后音频
- 对比原始音频与增强后音频的能量和试听效果

结果写法：

- 增强后背景噪声有一定下降
- 人声主体仍可保留
- 个别样本中存在轻微失真，但不影响后续识别与打断验证

### 6.2 实验二：STT 前后对比

目的：

- 验证轻量噪声抑制对识别结果的帮助

方法：

- 对 6 条单句样本分别做原始识别和增强后识别

结果写法：

- 在安静环境下提升有限
- 在风扇、水流等噪声环境下，增强后识别结果更稳定
- 说明噪声抑制对厨房场景具有一定价值

### 6.3 实验三：打断检测对比

目的：

- 验证自适应打断检测在小规模样本上的可行性

方法：

- 以 12 条打断样本评估有无降噪时的打断表现

结果写法：

- 正样本中大部分可被正确识别为打断
- 负样本中误触发数量可控
- 降噪后在复杂噪声条件下误打断有所下降，或总体更稳定

## 8. 论文写作建议

### 8.1 论文中如何描述实验规模

不要写：

- 大规模实验
- 全面优于现有方法
- 具有强泛化能力

建议写：

- 本文在自建小规模样本集上进行了可行性验证
- 结果表明该模块在厨房噪声环境下具有一定有效性
- 受时间和样本规模限制，后续将进一步扩展数据规模并进行更系统的评估

### 8.2 论文中如何描述创新点

建议表述：

1. 设计了一套面向厨房场景的自适应打断检测机制
2. 引入轻量级噪声抑制前端，提高嘈杂环境下的交互稳定性
3. 实现了从语音输入、打断检测、识别到回复输出的实时交互链路

### 8.3 论文中如何描述不足

可以直接写：

- 样本数量较少
- 场景覆盖仍有限
- 参数仍以经验调节为主
- 后续将扩展到更大样本和更多环境噪声条件
