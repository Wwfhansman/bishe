# 噪声抑制与语音增强设计文档

## 1. 文档定位

本文档用于同时支撑两类工作：

1. 后续代码开发和系统优化
2. 毕业论文中“复杂环境下语音前端优化”相关章节的写作

本文档的目标不是把噪声抑制吹成一个脱离工程实际的核心创新，而是清晰说明：

1. 厨房场景下为什么必须做噪声抑制
2. 当前项目已经具备哪些基础
3. 后续最可行的技术路线是什么
4. 在论文中应如何准确定位这一部分

## 2. 为什么噪声抑制值得写进论文

结论是：值得写，而且应该写。

原因有三点。

### 2.1 厨房环境本身就是强噪声场景

你的系统不是在安静书房里使用，而是在厨房使用。厨房噪声具有以下特点：

1. 持续噪声强
   - 油烟机
   - 风扇
   - 冰箱压缩机

2. 突发噪声多
   - 锅铲碰撞
   - 砧板切菜
   - 开关门

3. 非平稳噪声明显
   - 水流
   - 沸腾
   - 爆炒

这些噪声会直接影响：

1. STT 识别准确率
2. 打断检测稳定性
3. 声纹识别效果
4. 整体交互体验

### 2.2 噪声抑制是系统鲁棒性的重要组成部分

即使你的论文核心创新放在打断算法上，噪声抑制也仍然是重要的系统支撑模块。因为：

1. 打断检测输入来自麦克风信号
2. STT 输入来自麦克风信号
3. 声纹验证输入也来自麦克风信号

前端音频质量不稳，后面所有模块都会受影响。

### 2.3 这部分写得好，会让你的毕设更完整

很多论文只写“大模型、语音识别、问答”，但忽略真实场景。你如果把噪声抑制写清楚，会显得系统更真实、更像面向应用落地的毕设。

## 3. 论文中如何给这部分定位

### 3.1 正确定位

建议将这一部分定位为：

1. 语音前端增强模块
2. 面向厨房场景的鲁棒性交互优化
3. 实时语音助手的声学前处理策略

### 3.2 不建议的定位

如果你当前不自己实现完整降噪模型，就不要写成：

1. 提出了一种全新的噪声抑制算法
2. 提出了一种基于频率抵消的人声提取方法

这两种表述风险很大，尤其“频率抵消提取人声”技术上并不严谨。因为厨房噪声不是单一稳定频率，人声频带也和噪声频带大量重叠，简单做频率抵消并不能可靠提取目标人声。

### 3.3 推荐表述

建议在论文中使用以下表述：

1. 基于系统级 AEC/NS/AGC 的语音前端优化
2. 面向嘈杂厨房环境的语音增强与回声抑制设计
3. 面向语音交互鲁棒性的前端音频处理策略

## 4. 厨房场景中的噪声类型与挑战

### 4.1 背景噪声分类

可将厨房噪声分为三类。

#### A. 稳态噪声

特点：

- 持续时间长
- 频谱相对稳定

示例：

- 油烟机
- 风扇
- 冰箱

影响：

- 抬高背景噪声底线
- 降低低音量人声可检测性

#### B. 非稳态噪声

特点：

- 时变明显
- 振幅和频谱都会波动

示例：

- 水流声
- 沸腾
- 油炸爆裂声

影响：

- 使固定阈值方法失效

#### C. 瞬态冲击噪声

特点：

- 持续时间短
- 幅度突发高

示例：

- 锅铲碰撞
- 切菜
- 关柜门

影响：

- 容易被误判为用户打断

### 4.2 扬声器回声

除环境噪声外，你的系统还有一个非常关键的问题：TTS 播报会通过扬声器放出来，然后再次被麦克风采集。

这会造成：

1. STT 把 TTS 自己识别进去
2. 打断检测把 TTS 回声误认为用户说话
3. 声纹特征被污染

因此，回声问题必须和噪声问题分开分析，但在工程上又必须协同处理。

## 6. 可行的技术路线分析

### 6.1 路线一：仅依赖系统级 AEC/NS/AGC

做法：

1. 使用 Android/iOS 提供的系统级前处理
2. 不额外实现自定义降噪算法

优点：

1. 成本最低
2. 延迟最小
3. 工程集成简单

缺点：

1. 效果依赖设备
2. 不同机型表现不一致
3. 论文创新性较弱

适合定位：

- 工程优化基础版

### 6.2 路线二：系统级前处理 + 轻量语音增强

做法：

1. 前端继续保留系统级 AEC/NS/AGC
2. 在服务端或客户端增加轻量级语音增强模块

可选方法：

1. 谱减法
2. Wiener 滤波
3. 频谱门控
4. WebRTC NS

优点：

1. 工程可行
2. 算法可解释
3. 可以做前后对比实验

缺点：

1. 需要额外调参
2. 增益有限时需要谨慎表述

适合定位：

- 本科毕设最稳妥路线

### 6.3 路线三：深度学习语音增强

可选方法：

1. RNNoise
2. Demucs 类模型
3. DCCRN、FullSubNet 等深度增强模型

优点：

1. 理论效果更强
2. 论文上看起来更先进

缺点：

1. 集成成本高
2. 实时性与工程复杂度压力大
3. 数据和调试成本明显上升

适合定位：

- 后续扩展，不建议作为你当前毕设主线

## 7. 推荐技术路线

结合你当前项目基础，最推荐的路线是：

### 7.1 总体思路

采用三层式策略：

1. 客户端系统级回声消除与噪声抑制
2. 服务端鲁棒型打断检测
3. 可选的轻量语音增强模块

### 7.2 推荐落地方式

#### 第一层：保留并强化系统级 AEC/NS/AGC

继续使用当前 Flutter/Android/iOS 音频配置，不建议删除。

需要做的工作：

1. 验证不同设备上是否真的生效
2. 记录不同设备下效果差异
3. 在论文中明确说明这是“系统级前处理”

#### 第二层：继续强化服务端鲁棒性

即使前端已经做了降噪，服务端仍应保留：

1. 自适应噪声估计
2. 回声风险建模
3. 打断状态机

原因：

1. 前端降噪不是完美的
2. 回声问题很难完全依赖硬件解决

#### 第三层：补一个轻量增强模块

如果你想让这部分更有“算法工作量”，建议优先考虑：

1. 谱减法
2. Wiener 滤波
3. 简单频谱门控

这三种方法的优点是：

1. 原理清楚
2. 好写论文
3. 易于做前后实验对比

## 8. 不建议继续使用“频率抵消”的原因

你之前的设想是“通过语音频率的抵消去提取嘈杂环境下的人声”。这在论文里不建议这样写，原因如下：

1. 厨房噪声不是单频噪声
2. 人声和噪声频段高度重叠
3. 简单频率抵消会同时削弱人声
4. 实际效果很难稳定
5. 答辩时一旦被追问，很难自圆其说

更准确的表述应该是：

1. 语音增强
2. 噪声抑制
3. 回声消除
4. 声学前端优化

如果你确实做了频域方法，可以写成：

- 基于频谱分析的轻量语音增强方法

而不是：

- 通过频率抵消提取人声

## 9. 可实现的轻量算法方案

### 9.1 谱减法

基本思想：

1. 估计噪声频谱
2. 在频域上从混合语音频谱中减去噪声频谱
3. 重建增强语音

优点：

1. 原理经典
2. 易实现
3. 易写论文

缺点：

1. 易产生“音乐噪声”
2. 对非稳态噪声效果有限

适用性：

- 适合作为你毕设中的可实现增强算法

### 9.2 Wiener 滤波

基本思想：

1. 根据信噪比估计滤波增益
2. 抑制低信噪比频带

优点：

1. 比简单谱减更平滑
2. 数学表达清晰

缺点：

1. 仍依赖噪声估计质量

适用性：

- 适合作为谱减的替代或对照算法

### 9.3 频谱门控

基本思想：

1. 估计噪声门限
2. 低于门限的频谱区域强抑制

优点：

1. 实现简单
2. 效果直观

缺点：

1. 门限选择敏感

### 9.4 RNNoise

基本思想：

1. 使用轻量神经网络做实时降噪

优点：

1. 实际效果通常比传统方法好

缺点：

1. 集成复杂度更高
2. 本科毕设容易被拖慢节奏

## 11. 噪声抑制实验怎么做

### 11.1 评估目标

噪声抑制部分建议从两个角度评估：

1. 语音质量改善了多少
2. 对系统任务性能提升了多少

### 11.2 推荐任务型指标

由于你是语音助手项目，不一定必须追求复杂声学指标。更推荐任务导向指标：

1. STT 识别准确率提升
2. 打断误触发率下降
3. 打断漏检率下降
4. 主观听感更清晰

### 11.3 可选声学指标

如果你想更规范，也可以加入：

1. `SNR`
2. `Segmental SNR`
3. `STOI`
4. `PESQ`

但这些指标实现成本更高，不是必须项。

### 11.4 推荐实验设计

建议至少做两组对比：

1. 无额外增强，仅系统级 AEC/NS
2. 加入你实现的轻量增强算法

比较内容：

1. STT 识别正确率
2. 打断算法的误打断率
3. 打断算法的漏检率

这样能更直接说明噪声抑制对整个系统是有实际帮助的。

## 13. 后续开发建议

### 13.1 最小可行版本

如果你希望控制工作量，推荐的最小可行方案为：

1. 保留现有系统级 AEC/NS/AGC
2. 文档中明确这部分的作用和局限
3. 不立即引入深度学习降噪

这已经足够写进论文作为系统鲁棒性模块。

### 13.2 增强版本

如果你希望这部分也有明显代码工作量，建议下一步实现：

1. 一个轻量谱减法模块
2. 一组前后对比实验
3. 与打断算法联合评估

### 13.3 不推荐的方向

当前阶段不建议优先投入：

1. 重型深度降噪模型
2. 复杂多麦阵列处理
3. 纯“频率抵消”方案

这些方向工作量大、收益不稳定、答辩风险高。

## 14. 当前阶段结论

对于你的毕设，噪声抑制非常适合写进论文，但建议把它定位为：

1. 复杂环境下的语音前端优化模块
2. 提升 STT、打断检测和后续身份识别鲁棒性的支撑技术

最稳妥的技术路线是：

1. 保留系统级 AEC/NS/AGC
2. 继续依赖服务端的自适应噪声与回声鲁棒判断
3. 若时间允许，再补一个轻量语音增强算法

这样既符合工程实际，也方便后续扩展代码和论文写作。
